# Locode v0.5 Architecture Critique

**Date:** 2026-04-03
**Reviewer role:** Staff engineer / product-minded infra lead
**Subject:** v0.5 workflow automation — is this worth building, and how?

---

## 1. Executive Verdict

**Qualified yes, but you're one architectural mistake away from shipping an expensive wrapper.**

The core instinct — fixed state machines with LLM argument-filling, not free-form planning — is the most defensible idea in this entire proposal. Everything else is either unbuilt infrastructure that v0.5 assumes exists, a commodity that competitors will copy in a sprint, or a scope trap that will delay shipping.

The workflow automation direction moves Locode from "smart proxy" territory into something genuinely different: a **headless, checkpoint-driven engineering automation runtime**. That's a real product category. Claude Code, aider, Cursor, and Copilot Workspace are all interactive tools. None of them are designed to run unattended, pause at exactly the right moments, and resume cleanly. That gap is real and Locode can own it — but only if you don't bury the differentiation under a pile of half-built infrastructure.

The biggest risk isn't technical. It's that you ship v0.5 before v0.4 exists, build the ContextCurator on a foundation of grep and prayer, and discover that the "implement" step fails 40% of the time on anything non-trivial. Then the whole value prop collapses.

---

## 2. What Is Truly Differentiated

**2a. Fixed state machines over free-form planning**

This is the only genuinely non-copyable idea in the proposal, and it deserves to be the product's *core thesis*, not a design principle buried in a spec doc. Every other agent framework (LangGraph, AutoGen, CrewAI) still lets LLMs make control flow decisions. Locode says: no. The engine decides what happens next; the LLM only fills arguments. This is architecturally sound because small LLMs are bad planners but reasonable arg-fillers. It also makes the system auditable, testable, and debuggable in ways that planning-based agents are not.

**2b. Checkpoint-driven, human-in-the-loop automation by default**

Claude Code has no durable workflow state. Aider has no checkpoints. Copilot Workspace is one-shot. Locode's default posture — pause before push, pause before PR creation — is the right design for teams that can't afford silent mistakes in a shared repo. This is especially differentiated for engineering managers and team leads, not just individual developers. Nobody has shipped this well.

**2c. The cost model**

Local LLM for orchestration steps, Claude only for code generation. If this works at scale, a 5-ticket milestone might cost $0.15 in Claude tokens instead of $2-3 with a pure-Claude agent loop. That's a real number that justifies the complexity. Teams running sprints weekly will notice this.

**2d. Resumability as a first-class feature**

The ability to `Ctrl+C` and resume is underrated. Every other tool forces you to restart from scratch when something fails or you need to step away. Locode's disk-persisted state is the kind of thing developers will evangelize once they've been saved by it once.

---

## 3. What Is Weak, Commodity, or Easy to Copy

**3a. The YAML workflow templates**

Anyone can write `feature-ticket.yaml` in an afternoon. The template format itself is not a moat. The moat is the runtime that executes it reliably. Don't over-invest in template authoring ergonomics at the expense of engine robustness.

**3b. MCP integration**

Table stakes within 6 months. Cursor, Copilot, and Claude Code all have MCP support. The fact that Locode uses MCP for Linear and GitHub is good hygiene, not differentiation.

**3c. ContextCurator / context curation**

Every major coding tool is building this. Cursor has already shipped it. The differentiator isn't "we curate context" — it's *how reliably the curated context produces correct code*. That's a quality/eval story, not an architecture story. Don't pitch this as differentiated.

**3d. The original routing layer**

As frontier model costs drop and small models improve, the local-vs-Claude routing decision matters less. The 2024 case for routing was strong; the 2026 case is weaker. Don't let the routing layer (Locode's original core) become the product's identity. It's infrastructure, not positioning.

**3e. The progress display**

The terminal output mockup is nice but table stakes for any CLI tool with async steps. This is not a product decision — it's a polish task.

## 4. Biggest Architectural Risks

**4a. v0.4 doesn't exist yet, and v0.5 assumes it does**

The entire v0.5 value proposition depends on ContextCurator producing focused, high-quality bundles for Claude. ContextCurator depends on a working codebase indexer with symbol lookup and semantic search. That indexer is v0.4 — which is designed but not built.

Building v0.5's "gather_context" step without v0.4 means the local agent falls back to grep and read_file loops, which produce noisy context and mediocre implementations. The "walk away and come back to merged PRs" pitch requires the context to be *good*. It won't be if you skip v0.4.

**Ship v0.4 before touching v0.5's implement step. No exceptions.**

**4b. `Map<string, StepResult>` won't serialize**

```typescript
// In WorkflowInstance:
stepResults: Map<string, StepResult>
```

`JSON.stringify(new Map(...))` produces `{}`. Your StateManager calls `save()` which presumably serializes to disk. This bug will corrupt every workflow resume. Change it:

```typescript
stepResults: Record<string, StepResult>  // plain object, JSON-safe
```

**4c. The nested `for_each_ticket` step construct has no type support**

The YAML template defines:
```yaml
- id: for_each_ticket
  steps:
    - id: create_branch
    ...
```

But `WorkflowStepDef` has no `steps` field. The TypeScript types and the YAML templates are already inconsistent before a line of engine code is written. This means the engine will need to special-case loop steps with runtime `as unknown` casts — which is how you get hard-to-debug state corruption.

A loop step is a fundamentally different execution primitive from a regular step. Don't embed it as a magic YAML field. Model it explicitly (see §5).

**4d. `on_failure: fix_and_retry` is a goto in disguise**

```yaml
- id: run_tests
  on_failure: fix_and_retry

- id: fix_and_retry
  agent: coding-agent
  ...
```

This uses a sibling step ID as a jump target. This is not control flow — it's a goto. The engine will need to implement non-linear step execution, maintain a visited-step count to prevent infinite loops, and handle the case where `fix_and_retry` itself fails. None of this is in the type system.

Replace with a typed recovery strategy:

```typescript
type OnFailure =
  | { action: 'stop' }
  | { action: 'retry'; maxRetries: number }
  | { action: 'recover'; recoveryStepId: string; maxAttempts: number }
  | { action: 'continue' }  // mark failed, proceed anyway
```

**4e. CodingAgent's approval flow conflicts with workflow automation**

The existing CodingAgent (`coding-agent.ts`) is designed for interactive use: it shows diffs, asks for approval, presents plans. In workflow mode, `autoConfirm: true` bypasses all of this. But the approval layer isn't a boolean — it's the thing that makes Locode trustworthy. Silently auto-confirming everything in a workflow defeats the purpose of checkpoints.

You need a clean split: `CodingAgent` for interactive use, `WorkflowCodingAgent` (or a mode flag) that skips interactive prompts but still respects workflow-level checkpoints. These are different trust models.

**4f. No rollback for remote git state**

The v0.3.5 rollback model handles file edits. If a workflow pushes a branch and then fails mid-PR-creation, you have dangling remote state. "v0.5 assumes the v0.3.5 rollback model already exists" is accurate for local file state. It is silent about remote git state. Define what "undo" means for a partially-executed workflow before you ship `git push` automation.

## 5. Missing Abstractions

**5a. A first-class loop primitive**

The milestone workflow fundamentally requires iterating over a collection (tickets). This is not a workflow step — it's a control flow construct. Model it explicitly:

```typescript
type WorkflowStepDef = LinearStepDef | ForEachStepDef

interface ForEachStepDef {
  type: 'for-each'
  id: string
  iterateOver: string          // key in step results (must be an array)
  itemKey: string              // name to bind each item in sub-workflow
  subWorkflow: WorkflowTemplate  // inline or named reference
  checkpoint?: 'before-each' | 'after-each' | 'never'
  maxConcurrency?: number      // default 1 (sequential)
}
```

This makes the type system aware of loops, keeps the engine's control flow explicit, and avoids YAML magic nesting.

**5b. A typed step data bus**

Steps communicate via string keys (`output: tickets`, `input: tickets`), but the shape of what flows through is untyped. When the `implement` step receives `context_bundle`, is it a `ContextBundle`? A `GatheredContext`? A raw string? When the engine runs step 4 and step 3 failed with a partial output, what does step 4 see?

```typescript
interface StepOutput<T = unknown> {
  stepId: string
  data: T
  schema?: z.ZodType<T>  // optional, enables runtime validation
}
```

This doesn't need to be fully typed — but the engine should at least validate that required input keys exist before executing a step.

**5c. An approval context type**

When a checkpoint fires, what does the user see? The current spec has no approval UI definition for workflow mode. The existing REPL approval flow (`Do you want to proceed? [y/N]`) isn't sufficient — the user needs to see *what they're approving* in the context of the workflow.

```typescript
interface WorkflowApprovalRequest {
  workflowId: string
  stepId: string
  action: string             // human-readable: "Push branch feat/linear-12 to origin"
  preview?: string           // diff, PR title, etc.
  consequences: string[]     // ["Creates remote branch", "Triggers CI"]
  artifacts: string[]        // paths to files written so far
}
```

Without this, checkpoints are just blocking prompts with no context — which trains users to hit `y` without reading.

**5d. Workflow versioning / migration**

If a workflow is interrupted on step 3 and you update the template (adding a step between 2 and 3), resume breaks. The `StateManager` stores `template: string` (a name) but the actual template definition may have changed.

Pin the template definition at workflow start:

```typescript
interface WorkflowInstance {
  // ...
  snapshotTemplate: WorkflowTemplate  // full definition at start time, not just name
}
```

This makes resume deterministic regardless of template changes.

## 6. What the Killer Feature Should Really Be

The killer feature is not "automate your tickets." That's the pitch, but it's not the product truth. Copilot Workspace and Claude Code agents already attempt this.

**The killer feature is: you know exactly what happened and why, and you were in control the whole time.**

Locode's real value is **auditability + controllability at scale**. Every step logged. Every decision traceable. Every remote action gated. Resume from exactly where you stopped. The LLM never improvises control flow.

This is what engineering *teams* care about, not just individual developers. A senior engineer running `locode milestone start` and reviewing 4 PRs in the morning is a very different user than a solo developer chatting with Claude. That's a workflow automation tool for teams, not a chat enhancement.

Frame the killer feature as: **"Full visibility into every step, full control over every remote action, full auditability of every decision — with AI handling the hard parts."**

## 7. What to Change Before Building

**7a. Fix the types before writing any engine code**

The type inconsistencies identified in §4 will calcify into bugs. Fix these now:
- `Map<string, StepResult>` → `Record<string, StepResult>`
- Add `ForEachStepDef` as a discriminated union variant
- Replace `on_failure: string` with a typed `OnFailure` union
- Add `snapshotTemplate` to `WorkflowInstance`

**7b. Define the approval UX contract**

Write the `WorkflowApprovalRequest` type and stub the display logic before the engine is built. The checkpoint UX is not a polish task — it's the central user-facing interaction of the entire product.

**7c. Decide what "CodingAgent in workflow mode" means**

The current CodingAgent is interactive-first. Before wiring it into the workflow engine, decide: does it get a `WorkflowExecutionContext` that changes its behavior, or is there a separate `NonInteractiveCodingAgent`? The latter is cleaner. The former is faster. Either way, the decision needs to be made explicitly, not discovered at integration time.

**7d. Write a workflow engine integration test first (TDD)**

Before implementing `WorkflowEngine`, write a test that:
1. Starts a workflow with a 3-step template
2. Interrupts after step 2 (simulates Ctrl+C)
3. Loads state from disk
4. Resumes and completes step 3

This test will reveal every serialization and resume bug before you've written thousands of lines of engine code.

## 8. What to Cut or Delay in v0.5

**Cut from v0.5:**

- **`for_each_ticket` / milestone template** — requires the loop primitive you haven't built yet. Implement `single_ticket` first and prove it works. The milestone template is a v0.5.1 feature.
- **`update_ticket` step** — write-back to Linear/GitHub is high blast radius and low signal for proving workflow execution. Cut it. Users can update tickets manually in v0.5.
- **Parallel ticket execution** — mentioned as "future optimization." Delete the mention entirely from v0.5 docs. It's a distraction.
- **ContextCache** — the v0.4 context cache is sufficient. Don't add a new cache layer specific to workflow steps in v0.5.

**Delay to v0.5.1:**

- **Milestone template** (once single_ticket is validated)
- **`update_ticket` / Linear write-back** (once checkpoint UX is proven)
- **Custom workflow templates** (once built-ins are stable)

**What v0.5 should actually ship:**

1. `WorkflowEngine` with flat (non-looping) step execution
2. `StateManager` with JSON-safe serialization and resume
3. `single_ticket` template: gather_context → implement → run_tests → commit
4. Approval checkpoints before commit and before push
5. `locode implement --ticket LINEAR-42` and `locode implement "Add user auth"`
6. Real-time progress display
7. Error recovery: retry with error context, pause on rate limit

That's a shippable, demonstrable v0.5 that proves the core thesis without requiring the loop primitive, the milestone orchestration, or the Linear write-back.

## 9. Suggested Product Positioning

**Against Claude Code:**
Claude Code is a brilliant interactive pair programmer. Locode is what you run when you want to batch 4 tickets while you're in standup — and know exactly what it did and why when you come back.

**Against aider:**
Aider implements one change at a time from the command line. Locode runs multi-step engineering workflows end-to-end, checkpoints before every remote action, and resumes if interrupted.

**Against Cursor:**
Cursor is an interactive IDE assistant. Locode is headless batch automation — it runs in CI, in a tmux session, on a dev VM. No GUI required.

**Against Copilot Workspace:**
Copilot Workspace is one-shot generation inside GitHub. Locode is a configurable workflow runtime: you control what steps run, where checkpoints are, and what gets auto-approved. It also costs 80% less on repetitive work because the orchestration runs locally.

**Target persona:**
Senior engineers and tech leads who are comfortable with AI code generation but want *control* over automated workflows, not another chat interface. Teams running weekly sprints who want Friday-afternoon batch processing, not just in-IDE suggestions.

---

## 10. Revised One-Sentence Thesis

> **Locode is a checkpoint-driven workflow engine that automates ticket-to-PR pipelines by routing code generation to the best available model, giving engineers full auditability and control over every remote action.**

The original thesis ("measurable token savings") is still true but undersells the product. Token savings is a consequence. The thesis should be about control, auditability, and headless automation — the things no other tool provides.
